# Дизайн-документ: всестороннее улучшение zapret-next

## 1) Контекст и цели

`zapret-next` — это Windows-ориентированная сборка для обхода DPI-блокировок, основанная на `winws.exe` + WinDivert, с набором стратегий в `.bat`-скриптах и утилитой управления сервисом (`service.bat`).

### Бизнес- и продуктовые цели

- Повысить **процент успешных подключений** к целевым сервисам при меняющихся техниках DPI.
- Снизить **время до рабочего состояния** для нового пользователя (Time-to-First-Working-Strategy).
- Уменьшить число инцидентов: конфликтов с ПО, ложных «поломок», ручных ошибок при установке/обновлении.
- Сделать проект проще в сопровождении и безопаснее в эксплуатации.

### Нецели

- Переписывание движка `winws` (вне зоны данного репозитория).
- Полный отказ от `.bat` в краткосрочной перспективе.
- Добавление рискованных/непрозрачных бинарей без воспроизводимой цепочки поставки.

---

## 2) Текущее состояние (по репозиторию)

1. Управление реализовано в основном через batch-скрипты:
   - Стратегии (`general*.bat`) запускают `winws.exe` с длинными наборами правил.
   - `service.bat` реализует меню, установку службы, диагностику, update-check, переключатели и т.д.
2. Пользовательские списки доменов/IP создаются автоматически (`*-user.txt`).
3. Диагностика уже есть, но логика разрознена и частично эвристическая.
4. Версионирование/обновления частично автоматизированы (проверка версий в GitHub), но нет целостного канала управляемых обновлений конфигов/стратегий.

Это хороший практический baseline, но есть точки роста по архитектуре конфигураций, тестируемости, UX и операционной зрелости.

---

## 3) Ключевые проблемы и ограничения

### 3.1 Конфигурационная сложность

- Параметры стратегий повторяются между множеством `.bat`, трудно гарантировать консистентность.
- Любое изменение требует правки нескольких файлов и ручной синхронизации.

### 3.2 Хрупкость парсинга и сервисного режима

- `service.bat` парсит другие `.bat` для извлечения аргументов запуска.
- Для сложных quoting-кейсов это может давать трудноотлавливаемые баги.

### 3.3 Диагностика не структурирована

- Есть полезные проверки, но нет:
  - единого машинно-читаемого отчета,
  - приоритизации (critical/warn/info),
  - рекомендаций «следующего шага» в стандартизированном виде.

### 3.4 Отсутствие телеметрии качества стратегий (локальной, приватной)

- Пользователь не видит, какая стратегия стабильно лучше в его сети.
- Нет data-driven выбора (кроме ручного перебора).

### 3.5 DevEx и поддержка

- Нет «источника истины» для стратегий (DSL/JSON/YAML), из которого генерируются `.bat`.
- Ограниченная автоматическая валидация списков и синтаксиса конфигов.

---

## 4) Целевое видение

Сделать из проекта **управляемую платформу стратегий**, где:

1. Стратегии описываются декларативно (например, в YAML/JSON),
2. Скрипты запуска/службы генерируются автоматически,
3. Есть единый диагностический движок и структурированный репорт,
4. Пользователь получает мастер выбора/подбора стратегии,
5. Обновления списков и стратегий воспроизводимы, проверяемы и предсказуемы.

---

## 5) Архитектурные предложения

## 5.1 Единый формат стратегий (Strategy Spec)

Ввести каталог, например `strategies/*.yaml`, где каждая стратегия хранит:

- ID, название, назначение (YouTube/Discord/Games/универсальная),
- набор фильтров и параметров `winws`,
- зависимости от списков (`hostlist`, `ipset`),
- профили совместимости (Windows версия, known conflicts),
- приоритет/рекомендацию по умолчанию.

### Ожидаемый эффект

- Один источник истины.
- Уменьшение дублирования.
- Возможность генерации `general (ALT*).bat` и service-профилей автоматически.

## 5.2 Генератор launch-скриптов

Добавить утилиту (например, PowerShell/Python):

- Читает `strategies/*.yaml`.
- Генерирует:
  - `general*.bat`,
  - файл-индекс стратегий,
  - конфиг для `service.bat`.

Преимущество: ручная правка только спецификации, а не десятков батников.

## 5.3 Разделение `service.bat` на модули

Рефакторинг на:

- `scripts/service/core.bat` (меню/роутинг),
- `scripts/service/diagnostics.bat`,
- `scripts/service/install.bat`,
- `scripts/service/update.bat`.

Это снизит размер «монолита» и упростит code review.

## 5.4 Structured Diagnostics

Сделать диагностику двухуровневой:

1. **Человекочитаемый** вывод (как сейчас).
2. **JSON-отчет** (`logs/diagnostics-<timestamp>.json`) со структурой:

```json
{
  "timestamp": "...",
  "checks": [
    {"id": "bfe_running", "severity": "critical", "status": "pass"},
    {"id": "proxy_enabled", "severity": "warn", "status": "fail", "details": "..."}
  ],
  "recommendations": ["disable_proxy", "enable_doh"]
}
```

Это позволит лучше автоматизировать поддержку и triage issues.

## 5.5 Мастер выбора стратегии (CLI wizard)

При первом запуске:

- Проверить базовые предусловия (админ, WinDivert, DNS-over-HTTPS).
- Запустить быстрый smoke test нескольких стратегий.
- Зафиксировать «лучший профиль» и предложить установить его как service.

Ключевая метрика: снижение ручного перебора.

## 6) UX и продуктовые улучшения

1. **Quick Start Doctor**: единая команда `quickstart`, которая выполняет:
   - checks,
   - минимальную самонастройку,
   - запуск рекомендованной стратегии.

2. **Пояснимые ошибки**:
   - не просто «не запустилось», а «конфликт с service X, команда для исправления Y».

3. **Профили под сценарии**:
   - `video`, `voice-chat`, `gaming`, `balanced`, `aggressive`.

4. **Локальная история стабильности**:
   - время работы,
   - число перезапусков,
   - частота проблем.

5. **Улучшение документации для новичков**:
   - короткая матрица «симптом → вероятная причина → шаги».

---

## 7) Безопасность и доверие

1. **Прозрачность supply chain**:
   - хэши/подписи для бинарей,
   - скрипт `verify-integrity` для проверки целостности.

2. **Безопасный режим обновлений**:
   - разделить обновление «списков» и «бинарей»,
   - подтверждение перед заменой критичных файлов,
   - rollback на предыдущую рабочую конфигурацию.

3. **Минимизация привилегий**:
   - всё, что можно, выполнять без admin,
   - elevate только для сервисных операций.

4. **Журнал операций**:
   - что изменено (службы/реестр/файлы),
   - когда и каким скриптом.

## 8) Качество и инженерные практики

## 8.1 Валидация данных

Добавить CI/локальные проверки:

- валидность формата `strategies/*.yaml`,
- проверка дублей/битых доменов/IP,
- проверка существования referenced бинарей/листов.

## 8.2 Smoke-тесты

Автоматизировать:

- генерацию команд запуска каждой стратегии в «dry-run»,
- проверку, что команда корректно формируется,
- проверку service install/uninstall сценариев в test-shell.

## 8.3 Стандарты логирования

- Единые уровни: `INFO/WARN/ERROR`.
- Единый формат timestamp.
- Ротация логов по размеру/дате.

## 10) Метрики успеха

- **TTFW (Time to First Working)**: медиана времени до первой рабочей стратегии.
- **First-run success rate**: доля пользователей, у которых всё заработало без ручной правки.
- **Issue Deflection Rate**: доля проблем, решённых встроенной диагностикой.
- **Regression Rate** после обновлений списков/стратегий.
- **MTTR** (mean time to recovery) после «поломки» доступа.

---

## 11) Риски и как их снижать

1. **Рост сложности из-за нового слоя спецификаций**
   - Митигировать простым форматом и валидатором.

2. **Ломка обратной совместимости bat-флоу**
   - Сохранять legacy-режим в переходный период.

3. **Ложные срабатывания диагностики**
   - Явно маркировать confidence level проверок.

4. **Избыточная автоматизация может ухудшить контроль опытных пользователей**
   - Оставить «advanced mode» с полным ручным управлением.

## 13) Итог

В репозитории уже добавлены базовые quick wins из плана (документация структуры, скрипты целостности/валидации, команда doctor и базовое логирование сервиса), поэтому в этом документе оставлены только невыполненные улучшения.
